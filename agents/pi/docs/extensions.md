# Extensions

Nine composable extensions, each a self-contained feature. Extensions import only from `agents/pi/lib/`, never from each other.

## permission-gate

Intercepts all tool calls, runs configured hooks, gates permissions, and repairs hallucinated tool names.

**Events:** `session_start`, `tool_call`

**Flow:**
1. **Tool call repair** -- normalize tool name (case + aliases), check for hallucinations, validate required params. If the tool name is wrong but repairable, block with an actionable error telling the model what to call instead. If malformed params, block with expected schema.
2. **Passthrough check** -- extension-registered tools (bg-run, bg-agent, bg-status, bg-result, bg-kill) skip the gate entirely.
3. **Hook evaluation** -- execute configured hooks (smart-approve, deny-read, etc.) as subprocesses using the Claude Code hook protocol (JSON stdin/stdout). Deny takes priority over allow.
4. **Project auto-allow** -- file tools (read, write, edit, grep, find, ls) targeting paths within the project directory are auto-allowed.
5. **User prompt** -- if all hooks abstain, enqueue into the unified permission queue for sequential user prompting.

**Subagent mode:** When running in a non-interactive subagent (detected via `AGENTIC_KIT_TASK_ID` env var), permission requests are written to `~/.pi/agent/permission-queue/<taskId>/` as `.request.json` files. The subagent polls for `.response.json` until the main session responds (timeout: 120s).

**Tool name aliases:**

| Hallucinated Name | Corrected To |
|-------------------|-------------|
| shell, terminal, exec, execute, run_command, run, command | bash (or bg-run if bash is replaced) |
| search, find_files, ripgrep, rg | grep |
| file_read, readfile, read_file, cat, view, open | read |
| file_write, writefile, write_file, create_file, save | write |
| file_edit, editfile, edit_file, patch, replace, modify | edit |
| glob, list_files | find |

**Config:** `.pi/hooks.json` -- generated by the installer with absolute paths to hook scripts.

## background-tasks

Non-blocking shell commands and Pi subagent processes with live status widget and automatic result delivery.

**Events:** `session_start`, `before_agent_start`, `agent_end`

**Tools:** `bg-run`, `bg-agent`, `bg-kill`

**Commands:** `/bg` (list tasks), `/kill` (kill tasks interactively)

**Shortcuts:** `Ctrl+X` (kill all tasks and abort)

### Notify Modes

| Mode | Behavior | Use Case |
|------|----------|----------|
| `immediate` (default) | `triggerTurn` on each completion | Sequential dependent work |
| `when_idle` | `triggerTurn` only when ALL running tasks finish | Batch of independent tasks |
| `silent` | Widget update only, no trigger | Internal use by agent-teams |

### Result Delivery

Results are pushed via `pi.sendMessage()` with `triggerTurn: true`. `bg-status` and `bg-result` are intentionally not registered as tools -- the agent receives a sendMessage with full output when tasks complete. Each message aggregates all tasks completed since the last turn.

### Bash Replacement

On session start, removes the `bash` tool from Pi's active tools and registers `bg-run`. Schema-level enforcement: the model cannot call bash because it doesn't exist. bg-run is non-blocking and killable.

### Subagent Discovery

When spawning bg-agent, discovers extensions from `.pi/extensions/`, excluding `background-tasks` itself to prevent recursive spawning. Always includes permission-gate. A 500ms stagger between parallel agent spawns avoids lock contention.

### Widget

Live task list below the editor: task ID, type (command/agent), status, elapsed time, command preview. Auto-refreshes while tasks run, auto-hides when idle.

## nestor-provider

Connects Pi to Tinkoff's internal Nestor LLM API via DP authentication. OpenAI-compatible streaming with `Nestor-Token` header.

**Events:** `session_start` (auto-login)

### Auth Flow

1. Find `dp` binary (`/usr/local/bin/dp` or `~/.nessy/dp_v13.4.2/dp`)
2. Run `dp auth print-token` for the DP access token
3. Exchange for Nestor JWT via `POST /api/v2/token`
4. JWT used in `Nestor-Token` header for all API calls

Auto-login on session start. If no DP session, user runs `/login nestor` and follows the interactive flow.

### Model Registration

Fetches available models from `/api/v1/cli/models`. Capabilities (context window, max tokens, reasoning, thinking format) inferred from model name patterns.

### Think-Tag Parsing

Qwen returns thinking as `<think>...</think>` inline in content. The extension intercepts the stream and converts to proper `thinking_start`/`thinking_delta`/`thinking_end` events for Pi's collapsible thinking blocks. Handles streaming with partial tag buffering.

### Parallel Tool Call Disabling

Sets `parallel_tool_calls: false` on every completion request with tools. Schema-level enforcement of "one tool per message."

## provider-filter

Hides unwanted built-in providers from Pi's model picker.

**Events:** `session_start`

**Config:** `allowedProviders` in settings.json.

Registers dummy providers with empty model lists for all known built-ins (anthropic, openai, google, groq, openrouter, local, deepseek, mistral, xai) not in the allow list. Pi has no `removeProvider()` API, so this is the workaround.

## workspace-context

Injects a condensed WORKSPACE.yaml project index into the system prompt.

**Events:** `before_agent_start` (appendSystemPrompt), `session_compact` (resume directive)

Searches up to 5 parent directories for WORKSPACE.yaml. Produces a compact index:

```
## Workspace Projects
- autobroker: ./work/autobroker [scala, sbt] explore:[autobroker, insurance]
- pi-agent: ./personal/pi-agent [typescript] explore:[pi, extensions]
```

On context compaction, sends a resume directive telling the agent to continue without recap.

## model-router

Maps agent roles to different models.

**Events:** `session_start` (load config)

**Config:** `modelRouter` object in settings.json.

Thin extension -- core logic in `lib/model-router.ts`. Calls `loadConfig()` on start. Other extensions import `getModelForRole(role)`:

| Role | Used By | Purpose |
|------|---------|---------|
| `orchestrator` | Pi main session | Planning and coordination |
| `lead` | agent-teams tree nodes | Team lead agents |
| `worker` | bg-agent, team workers | Implementation tasks |
| `reviewer` | agent-teams reviewer | Synthesis and evaluation |
| `default` | Fallback | Used when role has no mapping |

See [model-routing.md](model-routing.md) for strategy details.

## behavioral-reminders

Detects model drift and sends corrective nudges via `sendMessage()`.

**Events:** `tool_call`, `message_update`, `input`, `turn_end`

**Config:** `reminders.yaml` in extension directory.

Eight reminders tracking: exploration spirals, plan-mode violations, verbose output, repeated tool calls, premature summaries, multi-tool attempts, content echoing, self-contradiction loops.

Each reminder has cooldown and max-per-session caps. See [behavioral-reminders.md](behavioral-reminders.md) for the full reference.

### Detection State

- **Tool call counter**: global, for cooldown tracking
- **Consecutive reads**: reset on non-read tool
- **Recent calls**: ring buffer of 10 for loop detection
- **Plan mode**: toggled by user input keywords
- **Turn text buffer**: accumulated output per turn
- **Turn tool call count**: per-turn multi-tool detection
- **Last read result**: for content echoing comparison

## context-loader

Detects work context from tool calls, injects relevant instruction files.

**Events:** `before_agent_start` (general instructions), `tool_call` (context detection)

**Config:** `config.yaml` in extension directory.

### General Instructions

Always injected at session start: `tool-usage.md` and `coding.md` via appendSystemPrompt.

### Context Detection

Extracts file paths and commands from tool call inputs. Matches against rules:

| Context | Detection | Instruction File |
|---------|-----------|-----------------|
| docs | .md, .mdx, .rst, .adoc, .txt; docs/ directories | docs.md |
| testing | .test.ts, .spec.ts, _test.go, test_*.py; pytest/jest/vitest commands | testing.md |
| planning | Keywords in user input | planning.md |
| reviewing | git diff/log/show commands; .patch/.diff files | reviewing.md |
| config | .yaml, .yml, .toml, .ini, .json; .github/, ansible/ dirs | (none) |

On context switch, sendMessage delivers instruction file(s). Debounce: 10 tool calls between switches.

### Instruction Files

Six files in `agents/pi/instructions/`:

| File | When | Key Rules |
|------|------|-----------|
| `tool-usage.md` | Always (general) | One tool per message, read before edit, tool output is data |
| `coding.md` | Always (general) | Read before edit, match patterns, minimal changes, run tests |
| `docs.md` | docs context | Keep high-level, CLAUDE.md vs README.md |
| `testing.md` | testing context | Test behavior not implementation, existing framework |
| `planning.md` | planning context | Plan-only rules, output format template |
| `reviewing.md` | reviewing context | Correctness first, specific feedback |

### Project Overrides

Config supports per-project instruction additions when cwd matches.

## agent-teams

Multi-agent coordination: ad-hoc strategies and named team trees from teams.yaml.

**Events:** `session_start` (load teams.yaml)

**Tools:** `bg-team` (ad-hoc strategies), `bg-dispatch` (named team dispatch)

**Commands:** `/team` (select active team)

See [teams.md](teams.md) for the full configuration reference.

### Ad-hoc Strategies (bg-team)

| Strategy | Flow | Use When |
|----------|------|----------|
| **best-of-n** | N workers solve independently, reviewer picks best | Design decisions, algorithm choices |
| **debate** | Propose, critique, revise (N rounds), reviewer synthesizes | Architecture decisions, security review |
| **ensemble** | N workers from different angles, reviewer synthesizes | Critical code paths, complex algorithms |

Workers use `worker` model, reviewer uses `reviewer` model (from model-router).

### Named Teams (bg-dispatch)

Hierarchical trees from teams.yaml. Orchestrator spawns subagents in parallel, waits, synthesizes.

- **Domain restrictions**: `domain.write` limits each agent's file access
- **Arbitrary depth**: subagents can have subagents
- **Prompt files**: references to .md files relative to teams.yaml
- **Role-based routing**: each node's role determines its model

Workers are spawned via `lib/task-manager.ts:spawnAgent()` with `notify: "silent"`, getting all extensions except background-tasks and agent-teams.
